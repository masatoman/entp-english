# 🚨 Lazy Loading 問題予防ガイド

## 📋 **問題の要約**

2025 年 9 月 18 日に発生した「真っ白ページ問題」の根本原因と予防策をまとめます。

## 🔍 **根本原因**

### **1. Lazy Loading の設定問題**

- **症状**: 全ての lazy loaded コンポーネントが「読み込み中...」で永続停止
- **原因**: Suspense の fallback 状態から抜け出せない
- **影響**: 15 ページが真っ白になる重大な問題

### **2. Export/Import 不整合**

- **症状**: "does not provide an export named" エラー
- **原因**: 名前付き export と default import の不一致
- **影響**: コンポーネントの読み込み失敗

### **3. Router 対応の不完全性**

- **症状**: "Cannot convert object to primitive value" エラー
- **原因**: props 形式のコンポーネントを Router 経由で呼び出し
- **影響**: ページ遷移時のクラッシュ

## 🛡️ **予防策**

### **A. Lazy Loading 使用ルール**

#### **✅ 推奨パターン**

```typescript
// 1. 直接import（推奨）
import Component from "../components/Component";

// 2. lazy loading使用時は必ずテスト
const Component = lazy(() => import("../components/Component"));
// → 必ずPlaywright MCPで動作確認
```

#### **❌ 禁止パターン**

```typescript
// lazy loading後のテストなし
const Component = lazy(() => import("../components/Component"));
// → 本番環境で突然動かなくなるリスク
```

### **B. Export/Import 統一ルール**

#### **✅ 推奨パターン**

```typescript
// コンポーネントファイル
export default function Component() { ... }

// import側
import Component from "./Component";
```

#### **❌ 禁止パターン**

```typescript
// 混在パターン（危険）
export const Component = ... // 名前付き
import Component from "..." // default import
```

### **C. Router 対応チェックリスト**

#### **新規コンポーネント作成時**

- [ ] `useNavigate()` を使用
- [ ] `useParams()` でパラメータ取得
- [ ] props 形式のコールバック関数を削除
- [ ] default export を使用

#### **既存コンポーネント修正時**

- [ ] props interface を削除
- [ ] onBack/onComplete 等を navigate 呼び出しに変更
- [ ] Router 経由での呼び出しに変更

## 🚨 **Git Push エラー対応ルール**

### **基本原則**

- **❌ 絶対禁止**: `git push --no-verify` の使用
- **✅ 必須**: 全てのエラーを修正してからプッシュ
- **✅ 必須**: プリコミットフックのエラーを尊重

### **`--no-verify` の重大なリスク**

#### **品質管理の破綻**

- テストエラーを無視してバグが混入
- 型安全性の破綻
- ビルド失敗のリスク

#### **チーム開発への悪影響**

- 他の開発者が壊れたコードを受け取る
- CI/CD パイプラインの失敗
- 開発効率の大幅低下

#### **長期的な技術債務**

- 問題が蓄積して後で大きな修正が必要
- テストの信頼性低下
- 保守性の悪化

### **例外的な使用条件（極めて限定的）**

#### **使用可能な条件**

1. **真の緊急事態**: システムダウン等の緊急対応時のみ
2. **一時的なデモ**: 動作確認のための一時的なプッシュ
3. **即座の修正約束**: 24 時間以内の確実な修正計画

#### **使用時の必須条件**

- 使用理由の明確な文書化
- 修正計画の具体的なタイムライン
- チーム全体への事前通知

### **正しい対応手順**

#### **エラー発生時**

1. **エラー分析**: プリコミットフックのエラー内容を詳細確認
2. **優先度判定**: 重要度に応じてエラーを分類
3. **段階的修正**: 重要なエラーから順次修正
4. **テスト実行**: 修正後の動作確認
5. **正常プッシュ**: エラーゼロでのプッシュ実行

#### **エラー分類**

- **🔴 重大**: 構文エラー・ビルドエラー → 即座修正必須
- **🟡 警告**: TypeScript エラー・未使用変数 → 計画的修正
- **🟢 軽微**: Lint エラー・フォーマット → 自動修正可能

### **品質保証の責任分担**

#### **開発者の責任**

- エラーを修正してからプッシュ
- テストの実行と確認
- コード品質の維持

#### **レビュアーの責任**

- エラーのあるコードを拒否
- 品質基準の維持
- 改善提案の提供

#### **システムの責任**

- プリコミットフックによる自動チェック
- CI/CD での品質ゲート
- エラーのあるコードのデプロイ防止

### **まとめ**

**`--no-verify` は「最後の手段」であり、基本的には使用禁止です。**

**品質を犠牲にした短期的な利便性は、長期的には大きな損失となります。**

**ENTP の特性である「効率性」を真に実現するためには、確実な品質管理が不可欠です。** 🎯
